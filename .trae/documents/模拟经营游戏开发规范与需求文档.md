# 模拟经营游戏开发规范与需求文档

## 1. 项目概述

### 1.1 项目背景

基于模拟经营（游戏策划）开发机制梳理，开发一款以「玩家成长（策划职级晋升）」为线索的模拟经营游戏，玩家将扮演游戏策划角色，完成从游戏基础设定到规模化运营的全流程。

### 1.2 技术栈

* 前端框架：Vue 3 + TypeScript

* 状态管理：Pinia

* 游戏引擎：Phaser 3

* 构建工具：Vite

* 样式框架：Tailwind CSS

* 包管理器：pnpm

* 代码质量：ESLint + Prettier

* 测试框架：Vitest

## 2. 开发流程

### 2.1 需求管理

1. 需求来源：产品经理提供的功能需求、开发机制梳理文档
2. 需求评审：每周进行需求评审会议，明确需求范围和验收标准
3. 需求变更：通过需求变更流程，评估影响范围并更新相关文档

### 2.2 开发流程

1. 任务分配：根据需求文档，将功能拆分为可执行的开发任务
2. 代码开发：遵循代码规范，编写高质量代码
3. 单元测试：为核心功能编写单元测试，确保代码质量
4. 代码评审：提交代码前进行代码评审，确保代码符合规范
5. 集成测试：将代码合并到主分支后进行集成测试
6. 验收测试：产品经理进行功能验收，确保符合需求

### 2.3 版本管理

1. 版本号规则：采用语义化版本号（MAJOR.MINOR.PATCH）
2. 分支策略：

   * main：主分支，用于发布正式版本

   * develop：开发分支，用于集成各功能模块

   * feature/xxx：功能分支，用于开发新功能

   * bugfix/xxx：bug修复分支，用于修复线上问题
3. 发布流程：从develop分支合并到main分支，打标签发布

## 3. 代码规范

### 3.1 基础规范

1. 使用TypeScript编写代码，确保类型安全
2. 遵循ESLint和Prettier配置，保持代码风格一致
3. 代码注释：为复杂逻辑、公共API添加注释
4. 避免重复代码，提取公共组件和工具函数

### 3.2 组件规范

1. 组件命名：使用PascalCase命名法，如`SimulationPanel.vue`
2. 组件拆分：遵循单一职责原则，将复杂组件拆分为多个子组件
3. 组件通信：优先使用Props/Events，避免过度使用Provide/Inject
4. 样式组织：使用Tailwind CSS，避免内联样式和复杂CSS

### 3.3 状态管理规范

1. Store命名：使用kebab-case命名法，如`game-store.js`
2. 状态设计：遵循单一数据源原则，避免冗余状态
3. Action命名：使用camelCase命名法，如`updateGameData`
4. 持久化：使用pinia-plugin-persistedstate管理持久化状态

### 3.4 命名规范

1. 变量命名：使用camelCase命名法，如`currentGameType`
2. 常量命名：使用UPPER\_SNAKE\_CASE命名法，如`MAX_PLAYERS`
3. 函数命名：使用camelCase命名法，如`calculateRevenue`
4. 类命名：使用PascalCase命名法，如`Simulation`

## 4. 目录结构规范

### 4.1 核心目录结构

```
src/
├── assets/          # 静态资源
├── components/      # Vue组件
├── composables/     # 组合式函数
├── constants/       # 常量定义
├── generators/      # 数据生成器
├── managers/        # 管理类
├── models/          # 数据模型
├── modules/         # 功能模块
├── router/          # 路由配置
├── stores/          # Pinia状态管理
├── styles/          # 全局样式
├── types/           # TypeScript类型定义
├── utils/           # 工具函数
├── App.vue          # 根组件
└── main.ts          # 入口文件
```

### 4.2 目录命名规则

1. 使用kebab-case命名法，如`game-config.js`
2. 组件目录：按功能模块组织，如`components/simulation/`
3. Store目录：按功能模块组织，如`stores/lottery/`

## 5. 质量标准

### 5.1 代码质量

1. ESLint检查：确保代码符合ESLint规则
2. 单元测试覆盖率：核心功能覆盖率≥80%
3. 代码复杂度：单个函数行数≤50行，避免深层嵌套

### 5.2 性能要求

1. 页面加载时间：首屏加载时间≤3秒
2. 交互响应时间：用户操作响应时间≤100ms
3. 内存使用：避免内存泄漏，定期进行内存监控

### 5.3 兼容性要求

1. 浏览器兼容：支持Chrome、Firefox、Safari最新版本
2. 设备兼容：支持桌面端和移动端

## 6. 协作机制

### 6.1 代码评审

1. 所有代码变更必须经过代码评审
2. 评审重点：代码质量、功能正确性、性能影响
3. 评审流程：使用GitHub Pull Request进行代码评审

### 6.2 文档协作

1. 使用Markdown编写文档，保持文档更新
2. 文档存储：与代码存储在同一仓库，便于版本管理
3. 定期更新：功能变更后及时更新相关文档

### 6.3 会议机制

1. 每日站会：15分钟，同步开发进度和遇到的问题
2. 每周例会：总结本周工作，规划下周任务
3. 需求评审：根据需求变更情况，灵活安排

## 7. 安全要求

### 7.1 数据安全

1. 敏感数据加密：用户敏感数据必须加密存储
2. 数据验证：前端和后端必须进行数据验证
3. 防止SQL注入：使用参数化查询，避免直接拼接SQL

### 7.2 前端安全

1. 防止XSS攻击：对用户输入进行过滤和转义
2. 防止CSRF攻击：使用CSRF令牌
3. 资源加载安全：使用HTTPS加载资源

### 7.3 依赖安全

1. 定期更新依赖：及时更新存在安全漏洞的依赖
2. 依赖审计：使用pnpm audit检查依赖安全

## 8. 需求文档

### 8.1 核心玩法需求

基于开发机制梳理文档，将游戏分为三个阶段：

#### 8.1.1 前期机制（新手阶段：0-7天）

1. **游戏基础设定**

   * 功能：选择游戏类型（MOBA、卡牌、休闲竞技）、设定游戏名称

   * 实现：创建游戏基础设定界面，支持类型选择和名称自定义

   * 验收标准：能够成功创建游戏，解锁核心数据面板

2. **首次基础策划操作**

   * 功能：发布普通皮肤、设置日常登录奖励、修复基础BUG

   * 实现：创建基础操作界面，支持导师引导

   * 验收标准：能够执行基础操作，查看数据变化

3. **游戏公开发布**

   * 功能：点击发布游戏，生成发布公告，开放下载渠道

   * 实现：创建发布游戏功能，模拟数据生成

   * 验收标准：能够成功发布游戏，查看初始数据

4. **新手目标达成**

   * 功能：完成累计下载量、市场情绪等目标

   * 实现：创建新手目标系统，实时跟踪进度

   * 验收标准：达成目标后解锁中期权限

#### 8.1.2 中期机制（成长阶段：8-30天）

1. **操作范围解锁**

   * 功能：分批次开放限定皮肤、英雄调整、渠道投放等操作

   * 实现：创建操作权限管理系统，根据天数解锁功能

   * 验收标准：能够按计划解锁高阶操作

2. **核心目标升级**

   * 功能：阶段性目标（月度收入、DAU峰值、付费率等）

   * 实现：创建目标管理系统，动态更新目标

   * 验收标准：能够查看和完成阶段性目标

3. **决策权衡设计**

   * 功能：操作消耗权衡、风险权衡、数据权衡

   * 实现：创建决策系统，模拟真实策划决策场景

   * 验收标准：操作结果符合预期，体现决策影响

#### 8.1.3 后期机制（成熟阶段：31天及以上）

1. **操作权限拉满**

   * 功能：解锁所有高阶操作，新增自定义操作

   * 实现：创建完整操作系统，支持复杂操作

   * 验收标准：能够执行所有高阶操作

2. **复杂挑战升级**

   * 功能：紧急事件升级、市场动态变化、长期目标多元化

   * 实现：创建动态事件系统，模拟复杂市场环境

   * 验收标准：能够应对各种复杂挑战

3. **职级晋升终极目标**

   * 功能：晋升策划总监，解锁终极权限

   * 实现：创建职级晋升系统，设定终极目标

   * 验收标准：能够达成策划总监目标

### 8.2 无限循环机制

1. **动态内容生成机制**

   * 功能：操作效果动态随机、挑战动态生成、游戏类型动态切换

   * 实现：创建动态生成系统，根据当前状态调整操作效果

   * 验收标准：操作效果根据环境动态变化

2. **内容迭代机制**

   * 功能：定期新增操作类型、季节/节日专属内容、数据维度迭代

   * 实现：创建内容迭代系统，支持定期更新

   * 验收标准：能够定期获取新内容

3. **策略多样性机制**

   * 功能：多运营路线可选、多游戏并行运营

   * 实现：创建策略系统，支持不同运营路线

   * 验收标准：能够选择不同运营路线，查看不同效果

4. **玩家自定义与社交激励**

   * 功能：全维度自定义、社交与排行榜激励

   * 实现：创建自定义系统和社交系统

   * 验收标准：能够自定义游戏规则，参与社交互动

## 9. 测试规范

### 9.1 单元测试

1. 测试范围：核心功能、数据模型、工具函数
2. 测试框架：Vitest
3. 覆盖率要求：核心功能≥80%
4. 测试文件命名：与被测试文件同名，后缀为.test.ts

### 9.2 集成测试

1. 测试范围：组件间交互、模块间集成
2. 测试工具：Vue Test Utils
3. 测试重点：数据流、状态管理、路由跳转

### 9.3 E2E测试

1. 测试范围：完整用户流程
2. 测试工具：Cypress（可选）
3. 测试重点：核心玩法流程、关键操作路径
