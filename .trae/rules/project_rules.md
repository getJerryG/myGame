## 开发环境
永远使用pnpm作为包管理器

## 开发流程
  1. 环境准备
     - 确保安装了Node.js和pnpm
     - 克隆仓库后执行 `pnpm install` 安装依赖
  2. 代码开发
     - 所有代码必须在pnpm环境下运行
     - 遵循现有的代码风格和架构模式
     - 使用Vue 3 Composition API进行开发
     - 2.1 模块划分：根据业务领域和功能边界，将系统明确划分为多个独立模块，每个模块需包含清晰的功能描述文档
    -   2.2 组件设计：采用组件化开发模式，将UI界面和业务逻辑拆分为可复用、低耦合的组件，每个组件需具备单一职责
    -   2.3 代码组织：建立规范的目录结构，按模块/功能/组件三级进行代码文件组织，确保文件路径与功能模块对应
    -   2.4 组件通信：定义清晰的组件间通信机制，明确父子组件、兄弟组件及跨模块组件的交互方式
    -   2.5 开发规范：制定组件开发标准，包括命名规范、接口定义、状态管理、样式隔离等具体要求
    -   2.6 文档要求：为每个模块和组件编写详细文档，包括功能说明、使用方法、接口定义及注意事项
  3. 代码质量检查
     -   执行 `pnpm lint <path>` 检查代码是否符合规范并对错误进行修复
     -   执行 `pnpm typecheck` 进行类型检查
     -   确保没有语法错误和警告
     -   确保没有未使用的变量、函数或导入
  4. 测试
     -   编写单元测试和集成测试
     -   执行 `pnpm test` 运行所有测试
     -   确保测试通过率为100%
     -   确保每个测试用例都有明确的断言，覆盖所有边界情况和异常处理
## 开发规范
  -  组件开发规范
     -   每个组件都必须有一个.vue文件，文件名必须与组件名相同
     -   组件的`<template>`部分必须有且只有一个根元素
     -   组件的`<script lang="ts" setup>`部分必须包含组件的逻辑代码，不能包含模板代码,使用Vue 3的Composition API，不能使用Vue 2的Options API
     -   组件的`<style scoped lang="scss">`部分必须包含组件的样式代码，不能包含全局样式
     -   组件的`<template>`部分必须使用Vue 3的模板语法，不能使用Vue 2的模板语法
    
## 目录使用规则
1. **别瞎建目录**：严格按上面的结构来，新增目录要和大家商量
2. **文件别乱放**：
   - 组件就放components里
   - 页面就放views里
   - 业务逻辑就放services里
3. **层级别太深**：最多4层目录，别搞成俄罗斯套娃

## 状态管理
- 用Pinia，别用Vuex
- 只有全局都要用的数据才放store里
- 每个功能模块建一个单独的store文件
- store里的状态名用小驼峰（比如`userInfo`）
- 方法名用动词开头（比如`getUserInfo`）

## 代码质量
### 1. 写完代码要检查
- 运行`pnpm lint`：检查代码格式，自动修复问题
- 运行`pnpm typecheck`：检查类型错误
- 确保没有红色警告和错误

### 2. 别写重复代码
- 重复超过2次的代码，就抽成函数或组件
- 通用功能放在utils目录里
- 别复制粘贴别人的代码，要理解后自己写

### 3. 注释要清晰
- 复杂逻辑要写注释，说明为什么这么做
- 函数要写注释，说明参数和返回值
- 别写没用的注释（比如`// 定义变量`）

### 4. 别用魔法数字
- 比如别直接写`if (status === 200)`，要定义常量`const SUCCESS = 200`
- 魔法数字放constants目录里
