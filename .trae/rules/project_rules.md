## 开发流程
 你收到的任何一个功能需求后，必须严格按照以下流程执行
1. **需求分析与拆分**
  - 确认需求规格和功能边界
  - 将需求拆分成多个功能点，每个功能点有明确的实现目标和验收标准
  - 把整个功能拆成多个独立、可完成、可验证的小任务
  - 每个任务只做一件小事（改一行、加一个函数、修一个逻辑都算单独任务）
  - 列出任务清单：任务 1、任务 2、任务 3…
2. **每个小任务必须走完整的流程**
  对每一个小任务，依次做：
  2.1 确认任务：明确本次改什么、做到什么效果
  2.2 修改代码
  2.3 同步文档
    2.3.1 代码变了 → 文档必须同步更新
    2.3.2 文档没有 → 补充进去
    2.3.3保证：文档 ↔ 代码 完全一致
  2.4 简单测试说明：怎么验证这一步是对的
  2.5 Git 提交
    2.5.1 改动文件：只提交本次任务相关的文件
    2.5.2 提交信息：使用Type前缀和任务描述，例如：`feat: 任务x 完成xxx`
    2.5.3 输出本次任务全部结果：包括代码、文档、测试结果等
  2.6 输出本次任务全部结果

3.  **Type说明**：
  - `feat`: 新功能
  - `fix`: Bug修复
  - `docs`: 文档更新
  - `style`: 代码格式（不影响功能）
  - `refactor`: 重构（非feat非fix）
  - `perf`: 性能优化
  - `test`: 测试相关
  - `chore`: 构建/工具/依赖更新
- **提交示例**：
  ```bash
  feat(hero): 新增英雄技能编辑功能
  
  - 实现技能树可视化编辑
  - 支持技能效果实时预览
  - 添加技能冲突检测
  
  Closes #123
  ```
- **提交频率**：每个功能点独立提交，禁止大量代码一次性提交
4.  强制要求
  - 不许跳任务、不许合并任务
  - 不许只改代码不改文档
  - 不许文档和代码不一致- 
  - 每一个小点，都必须走完：
    任务 → 代码 → 文档 → 测试 → Git → 输出

## 开发规范
  -  组件开发规范
     -   每个组件都必须有一个.vue文件，文件名必须与组件名相同
     -   组件的`<template>`部分必须有且只有一个根元素
     -   组件的`<script lang="ts" setup>`部分必须包含组件的逻辑代码，不能包含模板代码,使用Vue 3的Composition API，不能使用Vue 2的Options API
     -   组件的`<style scoped lang="scss">`部分必须包含组件的样式代码，不能包含全局样式
     -   组件的`<template>`部分必须使用Vue 3的模板语法，不能使用Vue 2的模板语法
    
## 状态管理
- 用Pinia，别用Vuex
- 只有全局都要用的数据才放store里
- 每个功能模块建一个单独的store文件
- store里的状态名用小驼峰（比如`userInfo`）
- 方法名用动词开头（比如`getUserInfo`）

## 代码质量
- 确保没有红色警告和错误

### 1. 别写重复代码
- 重复超过2次的代码，就抽成函数或组件
- 通用功能放在utils目录里
- 别复制粘贴别人的代码，要理解后自己写

### 2. 注释要清晰
- 复杂逻辑要写注释，说明为什么这么做
- 函数要写注释，说明参数和返回值
- 别写没用的注释（比如`// 定义变量`）

### 3. 别用魔法数字
- 比如别直接写`if (status === 200)`，要定义常量`const SUCCESS = 200`
- 魔法数字放constants目录里
