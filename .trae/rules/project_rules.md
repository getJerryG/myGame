## 开发环境
永远使用pnpm作为包管理器

## 开发流程

### 1. 需求分析与拆分
- 与产品经理或业务分析师合作，理解需求规格和功能边界
- **编写需求文档**：明确功能目标、用户场景、验收标准
- **技术方案设计**：
  - 评估技术可行性，识别技术风险点
  - 确定数据模型和接口设计
  - 评估性能要求（如列表数据量、并发用户数）
- **任务拆分原则**：
  - 将需求拆分为可独立开发、测试、交付的最小功能单元
  - 每个任务预估工作量（建议不超过8小时）
  - 明确任务优先级（P0阻塞/P1重要/P2优化）
  - 识别任务依赖关系，制定开发顺序

### 2. 开发前准备
- **创建功能分支**：从main分支创建 `feature/功能名称` 或 `fix/bug描述` 分支
- **接口契约确认**：
  - 与后端确认API接口定义（路径、参数、返回值）
  - 使用TypeScript定义接口类型，放置于 `types/` 目录
  - 如接口未就绪，先使用Mock数据开发
- **UI/UX确认**：
  - 确认设计稿或原型图
  - 复杂交互需与设计师确认动效和状态

### 3. 代码开发
- **开发顺序建议**：
  1. 先定义类型和数据结构
  2. 再开发Service层（业务逻辑）
  3. 最后开发UI组件
- **代码规范检查清单**：
  - [ ] 所有代码在pnpm环境下运行
  - [ ] 遵循Vue 3 Composition API + `<script setup>` 语法
  - [ ] 组件文件使用PascalCase命名（如 `GameReleaseApp.vue`）
  - [ ] 工具函数文件使用camelCase命名（如 `heroSkinManager.ts`）
  - [ ] 常量使用UPPER_SNAKE_CASE命名
  - [ ] 类型定义使用PascalCase命名
- **模块化开发要求**：
  - 3.1 模块划分：根据业务领域划分独立模块，每个模块包含功能描述文档
  - 3.2 组件设计：UI与业务逻辑分离，组件保持单一职责（不超过300行）
  - 3.3 代码组织：按模块/功能/组件三级组织，目录层级不超过4层
  - 3.4 组件通信：Props向下传递，Events向上通知，跨组件使用Pinia
  - 3.5 开发规范：遵循命名规范、接口定义、状态管理、样式隔离标准
  - 3.6 文档要求：复杂模块编写README，说明功能、用法、接口定义

### 4. 代码自查（提交前必做）
- **静态检查**：
  - [ ] 执行 `pnpm lint` 修复代码格式问题
  - [ ] 执行 `pnpm typecheck` 确保无类型错误
  - [ ] 检查无console.log、debugger等调试代码残留
  - [ ] 检查无未使用的变量、函数、导入
- **功能验证**：
  - [ ] 自测功能主流程是否正常
  - [ ] 验证边界情况（空数据、超长文本、网络异常）
  - [ ] 验证响应式布局（不同屏幕尺寸）
  - [ ] 检查内存泄漏（组件卸载时清理定时器、事件监听）

### 5. 代码审查（Code Review）
- **提交PR前准备**：
  - 确保分支基于最新main分支（先执行 `git pull origin main`）
  - 解决所有合并冲突
  - PR描述清晰说明：变更内容、原因、测试方式
- **审查关注点**：
  - 代码逻辑是否正确，有无明显bug
  - 是否符合项目架构和规范
  - 是否有重复代码可抽取
  - 性能是否有优化空间
  - 安全性（XSS、CSRF、敏感信息泄露）
- **审查通过后**：由审查者合并到main分支

### 6. 测试
- **测试策略**：
  - 单元测试：测试纯函数、工具类、Store逻辑
  - 集成测试：测试组件交互、页面流程
  - E2E测试：测试完整用户场景（关键路径）
- **测试要求**：
  - [ ] 核心功能必须有测试覆盖
  - [ ] 执行 `pnpm test` 所有测试通过
  - [ ] 测试覆盖率不低于80%
  - [ ] 每个测试有明确断言，覆盖正常和异常场景

### 7. 构建与部署
- **构建检查**：
  - [ ] 执行 `pnpm build` 生产构建成功
  - [ ] 检查构建产物大小，排查异常大包
  - [ ] 验证构建产物在本地预览正常
- **环境部署**：
  - 开发环境：自动部署feature分支
  - 测试环境：合并后自动部署，供QA测试
  - 生产环境：手动触发，需审批流程

### 8. Git提交规范
- **分支管理**：
  - `main`：生产分支，只接受合并请求
  - `feature/*`：功能开发分支
  - `fix/*`：Bug修复分支
  - `hotfix/*`：紧急修复分支（从main切出）
- **提交信息格式**（遵循Conventional Commits）：
  ```
  <type>(<scope>): <subject>
  
  <body>
  
  <footer>
  ```
- **Type说明**：
  - `feat`: 新功能
  - `fix`: Bug修复
  - `docs`: 文档更新
  - `style`: 代码格式（不影响功能）
  - `refactor`: 重构（非feat非fix）
  - `perf`: 性能优化
  - `test`: 测试相关
  - `chore`: 构建/工具/依赖更新
- **提交示例**：
  ```
  feat(hero): 新增英雄技能编辑功能
  
  - 实现技能树可视化编辑
  - 支持技能效果实时预览
  - 添加技能冲突检测
  
  Closes #123
  ```
- **提交频率**：每个功能点独立提交，禁止大量代码一次性提交

### 9. 迭代优化
- **技术债务管理**：
  - 定期重构老旧代码
  - 更新依赖版本，修复安全漏洞
  - 优化性能瓶颈
- **知识沉淀**：
  - 复杂问题记录解决方案
  - 更新项目Wiki和开发文档
  - 团队内部分享经验

## 开发规范
  -  组件开发规范
     -   每个组件都必须有一个.vue文件，文件名必须与组件名相同
     -   组件的`<template>`部分必须有且只有一个根元素
     -   组件的`<script lang="ts" setup>`部分必须包含组件的逻辑代码，不能包含模板代码,使用Vue 3的Composition API，不能使用Vue 2的Options API
     -   组件的`<style scoped lang="scss">`部分必须包含组件的样式代码，不能包含全局样式
     -   组件的`<template>`部分必须使用Vue 3的模板语法，不能使用Vue 2的模板语法
    
## 目录使用规则
1. **别瞎建目录**：严格按上面的结构来，新增目录要和大家商量
2. **文件别乱放**：
   - 组件就放components里
   - 页面就放views里
   - 业务逻辑就放services里
3. **层级别太深**：最多4层目录，别搞成俄罗斯套娃

## 状态管理
- 用Pinia，别用Vuex
- 只有全局都要用的数据才放store里
- 每个功能模块建一个单独的store文件
- store里的状态名用小驼峰（比如`userInfo`）
- 方法名用动词开头（比如`getUserInfo`）

## 代码质量
### 1. 写完代码要检查
- 运行`pnpm lint`：检查代码格式，自动修复问题
- 运行`pnpm typecheck`：检查类型错误
- 确保没有红色警告和错误

### 2. 别写重复代码
- 重复超过2次的代码，就抽成函数或组件
- 通用功能放在utils目录里
- 别复制粘贴别人的代码，要理解后自己写

### 3. 注释要清晰
- 复杂逻辑要写注释，说明为什么这么做
- 函数要写注释，说明参数和返回值
- 别写没用的注释（比如`// 定义变量`）

### 4. 别用魔法数字
- 比如别直接写`if (status === 200)`，要定义常量`const SUCCESS = 200`
- 魔法数字放constants目录里
